Integrating Data Insights with Retrieval-Augmented Generation (RAG)

Data Science & Engineering Review

Data Preparation & Cleaning (Task 1)

Missing Values

Missing values were handled primarily through **Time-based Interpolation**. This approach preserves the time series continuity, which is critical for sequence-dependent modeling.

Feature Scaling

All numerical features were **Standardized (Z-score scaling)**. This prevents variables with larger magnitudes (e.g., high temperatures) from dominating the distance-based clustering models.

Shutdown Detection

Data points corresponding to system shutdowns or idle periods were detected and **filtered out**. The analysis was specifically focused on **active operational data** (`active_data`) to prevent shutdown data from skewing state definitions.

Core Analysis Strategy (Task 1)

Operational State Clustering

**K-Means Clustering** was used to segment the scaled, active data into distinct operational modes (e.g., **Cluster 1: Normal Operation**, **Cluster 0: Low Efficiency**). This provides contextual baselines for subsequent analysis.

Contextual Anomaly Detection

The **Isolation Forest** model was run **separately within each cluster**. This contextual approach ensures true anomalies are detected by comparing data points only to their relevant operational state, rather than a global average.

Root Cause Analysis & Future Forecasting

**Local Z-Score Analysis** was applied to anomalies to pinpoint the most deviant variable ($\pm 30$ min window), identifying the potential root cause. This information feeds into our next step: developing **Predictive Forecasting Models** for state transitions and precursor signals.

Key Insights on Volatility (Task 1)

Anomaly Distribution

Total Anomalies Detected: 2949

  • Cluster 0 (Low Efficiency): **77** anomalies
  • Cluster 1 (Normal Operation): 2872 anomalies

Finding: **97.4%** of anomalies occur in the supposedly "Normal" state, indicating significant underlying volatility.

Root Cause Pattern

  • The most significant anomalies (highest Z-scores) are consistently driven by large deviations (spikes/drops) in **Draft variables** (`Cyclone_Outlet_Gas_draft`, `Cyclone_Inlet_Draft`).
  • This points to the **Draft Control Loop** and related physical equipment as the primary instability factor during normal operation.
  • Conversely, the Low Efficiency state (Cluster 0) is relatively stable, suffering fewer extreme events.

Conclusion & Recommendations (Task 1)

Actionable Conclusion

The "Normal Operation" state is not truly stable; its stability is frequently compromised by sudden, high-magnitude deviations in the Draft System. This constant state of micro-disruption must be addressed to improve overall process reliability and efficiency.

Recommendations for Task 2

  • **Prioritize Draft System:** Immediately focus engineering on investigating the Draft Control Loop hardware and logic.
  • **Real-Time Monitoring:** Implement a real-time **Local Z-Score alert** specifically for Draft variables.
  • **Predictive Focus:** Develop models to forecast the occurrence of Draft-related anomalies and the transition into the Low Efficiency state.

System Architecture: The RAG Pipeline (Task 2)

The RAG prototype connects unstructured technical documentation (PDFs) to a Large Language Model (LLM) for grounded, verifiable answers regarding system operation and maintenance.

1

User Query

"What causes a draft drop?"

2

Retrieval (FAISS)

Embed query, search vector index for top 'k' documents.

3

Augmentation

Format retrieved text as context for the LLM prompt.

4

Generation

LLM answers based only on provided context.

RAG Component Breakdown (Task 2)

Ingestion Pipeline

  • **Document Loader:** fitz (PyMuPDF) for robust PDF parsing.
  • **Chunking Strategy:** RecursiveCharacterTextSplitter to maintain semantic boundaries.
    • Chunk Size: 500 characters, Overlap: 100 characters.

Embedding & Indexing

  • **Embedding Model:** SentenceTransformer (all-MiniLM-L6-v2) chosen for speed and efficiency.
  • **Vector Database:** FAISS (IndexFlatL2) used for fast, in-memory vector similarity search (L2 distance).

Orchestration & Generation

Python functions manage the flow from embedding the query to searching the **FAISS index** to compiling the context. The final prompt integrates the retrieved content before passing it to the placeholder LLM for an answer.

RAG Challenge: Retrieval Strategy

The effectiveness of RAG hinges entirely on retrieving the most relevant context.

K-Value Optimization

  • The parameter **k** (retrieval_k=5) determines the number of chunks returned.
  • **Too small (low k):** Risk of missing crucial context.
  • **Too large (high k):** Risk of adding noise or overwhelming the LLM's context window.
  • **Solution:** Requires iterative testing with domain-specific queries to find the optimal balance.

Chunking Granularity

  • **Chunk Size (500) and Overlap (100)** are hyperparameters directly impacting retrieval success.
  • **Small chunks:** Better precision, but risk losing required surrounding context.
  • **Large chunks:** Better context, but lower precision/higher noise in retrieval.
  • **Solution:** Recursive text splitting helps preserve structure, but fine-tuning based on document structure (e.g., splitting by section/paragraph first) is recommended.

RAG Challenges: Hallucination Prevention & Scalability

Hallucination Guardrails

  • **Context Constraint:** The LLM prompt must strictly instruct the model to answer **ONLY** using the provided retrieved documents.
  • **Citation Enforcement:** Answers must include citations (source file and chunk ID) derived from the retrieved metadata to ensure verifiability.
  • **Fall-back Mechanism:** If retrieval yields no relevant results, the LLM should be instructed to respond "Information not found in documentation" rather than guessing.

Scalability Strategy

  • **Current:** FAISS is fast but operates **in-memory**, limiting the size of the documentation corpus.
  • **Future Scaling:** Transition to a persistent, distributed Vector Database (e.g., Pinecone, Milvus, or cloud-native options).
  • **Benefit:** Allows the RAG system to handle tens of thousands of engineering manuals and operational documents efficiently without memory constraints.
